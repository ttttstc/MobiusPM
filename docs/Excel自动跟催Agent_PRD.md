# 司环 MobiusPM  --  Excel-native 项目自动跟催 Agent PRD

版本：v1.0  
日期：2026-06-27  
适用阶段：第一阶段 MVP  
输入文件：`项目630流水线排期计划.xlsx`  
产品定位：基于现有 Excel 项目台账与 WeLink CLI 的项目事项自动扫描、自动跟催、回写与执行报告工具。

---

## 1. 背景

当前项目跟进主要通过 Excel 维护，工作簿中包含问题清单、GitCode issue、会议记录、迭代排期计划、关键里程碑、官方插件清单、流水线统计需求等内容。项目经理需要人工完成以下重复动作：

1. 每天查看哪些事项还未闭环。
2. 判断哪些事项超期、临期、阻塞、待验收、待排期。
3. 找到责任人并通过 WeLink 跟催。
4. 记录是否已催办、催办了几次、是否需要升级。
5. 汇总当前项目风险和待关注事项。

第一阶段目标不是重做项目管理系统，也不替代 Excel，而是在现有 Excel 上叠加一层自动化执行闭环：

```text
读 Excel → 识别待跟催事项 → 生成 WeLink 消息 → 自动发送 → 回写 Excel → 输出跟催报告
```

### 2.4 当前 Excel 对自动跟催的适配问题

| 问题 | 影响 | 设计处理 |
|---|---|---|
| 没有 WeLink ID | 无法直接调用 WeLink CLI 精准发送 | 增加 `责任人WeLinkID` 字段或独立联系人映射配置 |
| `问题状态` 与 `支持情况` 存在双状态 | `Open + 已完成`、`Close + 待验收` 等组合需要重新解释 | 增加标准化状态映射层，不直接用单字段判断 |
| 责任人字段存在多人混写 | 无法判断是逐个发送还是只发第一责任人 | 解析多责任人并支持 fanout 策略 |
| 计划时间大量为空 | 不能只基于日期判断是否跟催 | 同时基于优先级、支持情况、状态、说明判断 |
| 说明为多行非结构化文本 | 难以判断最后更新时间 | 第一阶段不解析说明时间线，新增 `最后更新时间` 字段 |
| 没有催办状态字段 | 无法去重、防骚扰、闭环 | 新增催办状态、催办次数、上次催办时间等字段 |
| 没有跟催日志 | 无法审计发送历史 | 新增 `FollowUpLog` Sheet 或本地 JSONL 日志 |
| 缺少数据质量校验 | 缺责任人、缺计划时间、缺 issue 不容易被发现 | 扫描时输出数据质量报告 |

## 3. 产品目标

### 3.1 第一阶段目标

构建一个最小可用的 Excel 自动跟催闭环工具，支持：

1. 读取现有 Excel，优先读取 `630攻关问题清单`。
2. 将 Excel 行标准化为内部 WorkItem。
3. 根据规则识别待跟催事项。
4. 基于责任人映射调用 WeLink CLI 自动发送提醒。
5. 对已发送事项进行 Excel 回写，记录催办状态。
6. 输出本次扫描和跟催报告。
7. 支持 dry-run，发送前预览待催清单与消息内容。

### 3.2 非目标

第一阶段不做：

1. 自然语言修改 Excel。
2. 自动读取 WeLink 回复并回写。
3. 大路标访谈和自动规划事项。
4. 深度 Project Audit。
5. 自动升级领导。
6. 多项目知识图谱。
7. 完整日报、周报、月报生成。

这些能力保留为第二阶段及以后演进。

## 5. 核心用户故事

### US-01：扫描 Excel 发现待催事项

作为项目经理，我希望系统每天读取 Excel 并告诉我哪些事项需要跟催，以减少人工筛选时间。

验收标准：

1. 能读取 `630攻关问题清单`。
2. 能识别 Open、待验收、开发中、待排期等状态。
3. 能识别缺责任人、缺计划时间、缺 WeLink ID 等异常。
4. 能输出待跟催清单。

### US-02：生成 WeLink 跟催消息

作为项目经理，我希望系统能根据不同跟催原因生成不同话术，而不是统一模板。

验收标准：

1. 待验收事项生成验收确认消息。
2. 开发中事项生成进展确认消息。
3. 待排期事项生成排期确认消息。
4. 超期事项生成超期提醒消息。
5. 缺少责任人的事项不发送，只进入异常报告。

### US-03：自动发送 WeLink 消息

作为项目经理，我希望系统能调用已有 WeLink CLI 自动发送提醒，降低人工跟催负担。

验收标准：

1. 支持 dry-run 模式，只预览不发送。
2. 支持 send 模式，真正调用 CLI。
3. 缺少 WeLink ID 的事项不发送。
4. 同一事项 24 小时内不重复发送。
5. 同一责任人每天发送数量不超过配置阈值。

### US-04：回写 Excel 催办状态

作为项目经理，我希望发送成功后 Excel 自动记录催办状态，便于后续追踪。

验收标准：

1. 成功发送后写回 `催办状态=已催办`。
2. `催办次数` 自动 +1。
3. `上次催办时间` 更新为当前时间。
4. `最近跟催原因` 写入本次触发规则。
5. 保留原始字段，不破坏现有表结构。

### US-05：输出本次执行报告

作为项目经理，我希望每次运行后看到扫描结果、发送结果和失败原因。

验收标准：

1. 输出扫描总数、待催数量、发送成功数量、发送失败数量。
2. 输出未发送原因：缺责任人、缺 WeLink ID、频控拦截、状态无需跟催。
3. 输出 Markdown 报告。
4. 可选输出 JSON 报告供后续系统使用。

```text
启动命令 / 定时任务
        ↓
读取 Excel 工作簿
        ↓

        ↓

        ↓

        ↓

        ↓
生成 ReminderCandidate
        ↓

        ↓

        ↓

        ↓

        ↓
记录 FollowUpLog
        ↓

```

## 8. 标准状态映射

当前 Excel 同时存在 `问题状态` 和 `支持情况`，第一阶段需要映射为系统内部标准状态。

| 问题状态 | 支持情况 | 标准状态 | 跟催含义 |
|---|---|---|---|
| Open | 待验收 | 待验收 | 需要责任人确认验收进展或关闭条件 |
| Close | 待验收 | 待验收待关闭 | 可能已经关闭问题，但仍需验收确认 |
| Open | 已完成 | 完成待关闭 | 需要确认是否可以 Close |
| Close | 已完成 | 已关闭 | 不跟催 |
| Open | 开发中 | 开发中 | 需要进展跟催 |
| Open | 待排期 | 待排期 | 需要确认排期和计划时间 |
| Open | 挂起(630后分析) | 挂起 | 默认不自动催办 |
| Open | 重复单 | 重复 | 默认不自动催办 |
| Open | 拒绝 | 拒绝 | 不催办 |
| 任意 | 拒绝 | 拒绝 | 不催办 |

优先级映射：

| Excel 优先级 | 系统优先级 | 默认跟催策略 |
|---|---|---|
| 必备(630前) | P0 | 自动催办 |
| 增强(争取630) | P1 | 自动催办或草稿，可配置 |
| 长期演进(630后) | P2 | 默认不自动催办，只进报告 |
| 拒绝(不处理) | Ignore | 不催办 |

## 9. 跟催规则设计

### 9.1 数据质量规则

| 规则ID | 规则 | 处理动作 |
|---|---|---|
| DQ-001 | 活跃事项缺少责任人 | 不发送，进入异常报告 |
| DQ-002 | 活跃事项缺少 WeLink ID | 不发送，进入异常报告 |
| DQ-003 | P0/P1 活跃事项缺少计划时间 | 生成“补充计划时间”提醒 |
| DQ-004 | 计划时间无法解析 | 进入异常报告 |
| DQ-005 | 责任人字段包含多人但无法解析 | 不自动发送，进入人工确认 |

### 9.2 自动跟催规则

| 规则ID | 触发条件 | 跟催类型 | 默认是否发送 |
|---|---|---|---|
| R-001 | P0/P1 且标准状态=待验收 | 验收确认 | 是 |
| R-002 | P0/P1 且标准状态=完成待关闭 | 关闭确认 | 是 |
| R-003 | P0/P1 且标准状态=开发中，且计划日期早于今天 | 超期进展 | 是 |
| R-004 | P0/P1 且标准状态=开发中，且计划日期距离今天 <= 2 天 | 临期进展 | 是 |
| R-005 | P0/P1 且标准状态=待排期 | 排期确认 | 是 |
| R-006 | P0/P1 且缺少计划时间 | 计划补齐 | 是 |
| R-007 | 标准状态=挂起/重复/拒绝/已关闭 | 不催办 | 否 |
| R-008 | 同一事项 24 小时内已发送同类跟催 | 频控拦截 | 否 |
| R-009 | 同一责任人当天已达上限 | 频控拦截 | 否 |

### 9.3 当前 Excel 下的第一批高价值扫描口径

当前 Excel 中，很多事项缺少计划时间，因此第一版不应只依赖 `计划时间`。建议第一优先级扫描：

1. `必备(630前) + Open + 待验收`：需要催责任人确认是否验收通过。
2. `必备(630前) + Open + 已完成`：需要催是否可以关闭。
3. `必备(630前) + Open + 开发中`：需要催当前进展和完成时间。
4. `必备(630前) + Open + 待排期`：需要催排期。
5. `增强(争取630) + Open + 待验收/开发中/待排期`：可按配置自动催或生成草稿。
6. `活跃事项 + 缺责任人/缺 WeLinkID/缺计划时间`：不发消息，进入数据质量报告。

## 10. WeLink CLI 集成设计

### 10.1 CLI 封装

系统不直接散落拼接命令，应封装统一接口。

```ts
export interface WeLinkNotifier {
  sendMessage(input: {
    to: string
    message: string
    itemId: string
    runId: string
  }): Promise<{
    success: boolean
    messageId?: string
    error?: string
  }>
}
```

底层可以适配已有 CLI，例如：

```bash
welink send --to <welink_id> --message "<message>"
```

实际命令参数以已有 WeLink CLI 为准，PRD 只定义适配层职责。

### 10.2 消息发送策略

| 场景 | 策略 |
|---|---|
| 单责任人 | 直接发送给该责任人 |
| 多责任人 | 默认发送给第一责任人，抄送能力后续再做；也可配置逐个发送 |

## 11. 去重与频控设计

### 11.1 幂等 Key

每次发送前生成幂等 Key：

```text
{excel_file_hash}:{sheet_name}:{item_id}:{reminder_type}:{date}
```

同一天同一事项同一跟催类型只发送一次。

### 11.2 频控规则

| 规则 | 默认值 |
|---|---:|
| 同一事项同类跟催冷却时间 | 24 小时 |
| 同一责任人每天最多自动消息数 | 5 条 |
| 单次运行最大发送总数 | 50 条 |
| 缺少 WeLink ID 是否发送 | 否 |
| P2 是否自动发送 | 否 |

### 11.3 发送模式

| 模式 | 说明 |
|---|---|
| dry-run | 只扫描、生成消息和报告，不发送、不回写 |
| send | 执行发送，发送成功后回写 Excel |
| send-with-confirm | 生成候选清单，人工确认后发送；可作为后续聊天入口能力 |

## 12. Excel 回写设计

### 12.1 回写原则

1. 不覆盖原始业务字段。
2. 不删除用户已有内容。
3. 新增字段靠右追加。
4. 发送成功后才回写催办状态。
5. 回写失败时输出错误并保留发送日志。
6. 每次写入前自动备份原 Excel。

### 12.2 回写字段

发送成功后更新：

| 字段 | 更新规则 |
|---|---|
| 催办状态 | 写入 `已催办` |
| 上次催办时间 | 当前时间 |
| 催办次数 | 原值 + 1 |
| 最近跟催原因 | 写入触发规则，如 `R-001 待验收确认` |
| 最近跟催消息 | 写入消息前 200 字摘要 |

### 12.3 FollowUpLog 记录

每条发送记录都写入 FollowUpLog：

```json
{

  "reminderType": "acceptance_confirm",
  "message": "...",
  "sendStatus": "success",
  "sentAt": "2026-06-27T09:00:00+09:00"
}
```

## 17. 验收标准

### 17.1 Excel 读取验收

1. 能正确读取 7 个 Sheet。
2. 能识别 `630攻关问题清单` 为主数据源。
3. 能读取 177 条问题数据。
4. 能正确识别以下字段：序号、用户问题原文、问题状态、来源、优先级、责任人、支持情况、计划时间、说明。
5. 能将 Excel 日期序列号转为标准日期，例如 46188 转为 2026-06-15。

### 17.2 规则扫描验收

1. 能识别 `必备(630前) + Open + 待验收`。
2. 能识别 `必备(630前) + Open + 已完成`。
3. 能识别 `必备(630前) + Open + 开发中`。
4. 能识别 `必备(630前) + Open + 待排期`。
5. 能识别缺责任人、缺计划时间、缺 WeLink ID。
6. 默认不催办挂起、拒绝、重复、已关闭事项。

### 17.3 WeLink 发送验收

1. dry-run 模式不调用 CLI。
2. send 模式调用 CLI。
3. CLI 成功返回后记录 success。
4. CLI 失败返回后记录 failed 和错误信息。
5. 缺 WeLink ID 不发送。
6. 频控命中的事项不发送。

### 17.4 Excel 回写验收

1. 自动备份原 Excel。
2. 如果系统字段不存在，自动追加。
3. 发送成功后更新催办状态、上次催办时间、催办次数、最近跟催原因。
4. 不覆盖原始说明字段。
5. FollowUpLog 记录完整。

## 19. 边界与安全

1. 默认先使用 dry-run，确认规则正确后再开启 send。
2. 第一次运行建议只对少量责任人白名单发送。
3. 所有自动发送动作必须可追溯。
4. 不允许自动删除 Excel 行。
5. 不允许自动修改原始问题状态。
6. 回写仅限系统字段。
7. WeLink 消息不得包含敏感凭据、token、密钥。
8. CLI 参数需要做 shell escaping，避免命令注入。
9. Excel 写入失败时不能重复发送消息。
10. 同一事项必须具备幂等发送保护。

## 23. 结论

基于当前 Excel 的真实结构，第一阶段应优先落地“Excel 自动跟催 Loop”，而不是先做完整项目管理 Agent。原因是：

1. 当前 `630攻关问题清单` 已经具备事项、状态、优先级、责任人、处理方、计划时间、说明等核心字段。
2. 当前存在大量 `待验收`、`已完成但 Open`、`待排期`、`缺计划时间` 的事项，自动跟催价值明显。
3. 已有 WeLink CLI，可直接形成行动闭环。
4. 只需追加少量系统字段，即可在不破坏现有 Excel 工作方式的前提下实现自动化。

第一阶段 MVP 的一句话定义：

> 基于现有 Excel 项目台账，自动识别 630 前高优先级待闭环事项，通过 WeLink CLI 自动跟催责任人，并将催办状态回写 Excel 的轻量项目管理闭环工具。
