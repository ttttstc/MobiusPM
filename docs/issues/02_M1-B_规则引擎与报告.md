# [M1-B] 跟催规则引擎 + HTML/JSON 报告(dry-run 看板)

## 目标
基于 M1-A 产出的 WorkItem,实现数据质量规则 + 跟催规则,生成候选清单与本次扫描报告。这是**第一次见价值的节点**:跑完即可看到"今天该催的事项",验证规则准确性。仍是 dry-run,**不发送、不落催办状态**。

## 范围
- 数据质量规则 DQ-001 ~ DQ-005(PRD §9.1 全表)
- 跟催规则 R-001 ~ R-009(PRD §9.2 全表)
- `ReminderCandidate` 生成(PRD §15.2 数据结构)
- 幂等 key 公式(PRD §11.1):`{excel_file_hash}:{sheet_name}:{item_id}:{reminder_type}:{date}`
- HTML 报告生成(单文件,可独立浏览器打开)
- JSON 报告生成(供后续系统/diff 使用)
- 报告归档目录 `state/reports/{ts}/`

## 设计要点(已定决策)
1. **不做发送、不做去重历史**:本 issue 仅生成"如果要发该发什么"的候选清单。频控和幂等的**存储**在 M2-A 实现,本 issue 只生成 `dedupe_key` 字段。
2. **报告 = MVP 的看板**:HTML 单文件,4 个区块:
   - 本次扫描摘要(总数/活跃/可发/跳过/失败)
   - 待催明细表(事项ID/优先级/标准状态/责任人/跟催原因/规则ID)
   - 数据质量异常(缺责任人/缺计划/缺 WeLink ID/无法解析)
   - 失联事项(本次扫描后实现 vanished 检测的占位,M2-A 真正接入)
3. **报告样式**:纯 HTML + 极简 CSS,不引外部 JS/CSS,离线可用。
4. **看板就是 HTML 报告**:不做本地 Web 服务(收敛决策 D6)。
5. **PRD 6.30 数据下的"首批 5 类口径"**(PRD §9.3)必须能在报告里清楚区分。

## 输入 / 输出
**输入:** WorkItem 列表(M1-A 产出)、`config/pm-agent.yaml`
**输出:**
- `state/reports/{ts}/report.html` — 浏览器打开看
- `state/reports/{ts}/report.json` — 机器可读
- `state/reports/{ts}/candidates.yaml` — 候选清单(M2-B 的输入)

## 验收标准
1. ✅ `pm-agent scan` 跑完产出三个文件
2. ✅ 报告中活跃候选数与现状分析吻合(77 ±5)
3. ✅ DQ 规则的 5 种异常各至少 1 条在样例数据中能被识别出
4. ✅ R 规则的 6 类活跃跟催(R-001~R-006)在样例数据中均有匹配
5. ✅ 标记为 `挂起/重复/拒绝/已关闭` 的事项**不进**候选清单(R-007)
6. ✅ HTML 报告离线打开正常,4 个区块齐全
7. ✅ JSON 报告字段与 PRD §15.2 `ReminderCandidate` 对齐
