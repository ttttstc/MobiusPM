# [M2-A] 外置状态存储(SQLite)+ 幂等 + 频控去重

## 目标
把催办状态、发送日志全部外置到 SQLite,实现**主表不写、重跑不重发、单人不轰炸**的三重保障。本 issue 完成后,M1-B 的报告会自动带上"重复候选已被频控拦截"的标注,但仍未真正发送。

## 范围
- SQLite 初始化与 schema 创建(本仓库设计文档 §7 两张表)
- `store.py`:封装查询/写入接口
- 幂等检查:命中已有 `success/dry_run` 记录则跳过
- 频控规则实现:
  - 同一事项同类跟催冷却 24h
  - 同一责任人每天 ≤ 5 条
  - 单次运行最大 50 条
- 失联检测:本次未见的旧 itemId 标 `vanished_at`
- M1-B 报告新增"已被频控拦截"区块

## 设计要点(已定决策)
1. **SQLite,不是 jsonl**:理由——单文件零运维 + 索引查询比 jsonl 高几个数量级 + 未来扩展性最好。
2. **append-only follow_up_log**:历史记录永不更新,只插入。`item_state` 是事项级聚合表,每事项一行。
3. **幂等 key 已在 M1-B 生成**,本 issue 只做存储和查询。
4. **频控决策在 `candidate.py` 完成**:把"应发但被拦截"的候选标 `blocked=true` + `blocked_reason`,**仍出现在报告里**(透明可审计),不进入 send。
5. **失联事项**保留 30 天,30 天后归档/清理(本 issue 不做清理,留 TODO)。
6. **DB 路径默认 `state/pm-agent.db`**,git 忽略。
7. **并发安全**:SQLite WAL 模式开启;但 MVP 单进程跑,不并发,WAL 仅作未来兜底。

## 输入 / 输出
**输入:** M1-B 产出的 `candidates.yaml`,以及历史 `pm-agent.db`
**输出:**
- `pm-agent scan` 后 `candidates.yaml` 中部分候选标 `blocked: true + blocked_reason`
- `pm-agent.db` 中 `item_state` 表更新 `last_seen_at` + vanished 标记
- 不写入 `follow_up_log`(那是 M2-B send 时的事)

## 验收标准
1. ✅ 首次跑 `scan`,自动创建 `state/pm-agent.db` 与两张表
2. ✅ 同一 Excel + 同一天连跑两次 `scan`,第二次报告里所有候选 `blocked_reason=cooldown_24h` 或被 dedupe_key 命中
3. ✅ 人为构造单责任人 6 条 P0 候选,第 6 条 `blocked_reason=owner_daily_limit`
4. ✅ 候选总数超过 50,超出部分 `blocked_reason=run_limit`
5. ✅ 改一条事项的"用户问题原文",该 item 上次的 itemId 在 `item_state` 标 `vanished_at`,新 itemId 进入新行
6. ✅ 全流程主 Excel 文件 mtime 不变
