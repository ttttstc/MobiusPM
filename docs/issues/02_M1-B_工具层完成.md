# [M1-B] 工具层 Part 2:规则建议器 + Notifier + 消息渲染 + ask_human / write_decision

## 目标
补齐工具层剩下的全部 tool:规则建议(降级为 LLM 可参考的工具)、消息渲染、Notifier(含 mock)、人机确认接口、决策日志写入。本 issue 完成后,**所有工具就绪,可以独立组合跑通"读表 → 规则建议 → 渲染消息 → 模拟发送 → 写决策"全流程,只差 agent loop**。

## 范围
- `tools/rules.py`:
  - `query_rule_suggestions(work_items: list) → suggestions` tool
  - 实现 PRD §9.1 全部 DQ 规则 + PRD §9.2 全部 R 规则
  - 输出格式:`[{item_id, rule_id, reminder_type, severity, rationale_hint}, ...]`
  - **强调**:这是"建议",LLM 可采纳/补充/拒绝,不是硬性指令
- `tools/messages.py`:
  - `gen_message(item_id, reminder_type, context?) → message_text` tool
  - 实现 PRD §10.3 四个模板:acceptance_confirm / progress_check / schedule_confirm / due_date_missing
  - 占位符未填值显示"未填写"
  - 长度限制(可配置,默认 1000 字)
- `tools/notifier.py`:
  - `WeLinkNotifier` 接口(协议)
  - `MockNotifier` 实现:
    - 默认返回 success
    - 把"发送内容"追加写 `state/mock_sent.jsonl`
    - 可配置故意失败(用于测试 P2 安全边界)
  - `send_welink(item_id, owner, welink_id, message, reminder_type, confirmation_token) → result` tool
  - **tool 内部强制安全检查**(P2 原则):
    1. 查 `follow_up_log` 是否已有同 dedupe_key 的 success/mock 记录 → 命中返回 `{status: 'blocked', reason: 'dedupe'}`
    2. `check_rate_limit(welink_id)` → 命中返回 `{status: 'blocked', reason: 'rate_limited'}`
    3. `check_run_limit(run_id)` → 命中返回 `{status: 'blocked', reason: 'run_limit'}`
    4. 联系人不在 enabled 白名单 → `{status: 'blocked', reason: 'not_whitelisted'}`
    5. `confirmation_token` 校验:必须匹配最近一次 `ask_human` 返回的 token,否则 `{status: 'error', reason: 'no_confirmation_token'}`
  - 通过后调 Notifier 实际发送,写 `follow_up_log`,更新 `item_state`
- `tools/human.py`:
  - `ask_human(candidates: list, question: str) → {confirmation_token, confirmed_item_ids, edited_messages?}` tool
  - MVP 实现:**stdin 交互**——把候选打印到终端,让 PM 输入要催的事项 ID(用逗号分隔),可选输入修改后的消息
  - 返回的 `confirmation_token` = uuid,写入内存(本次 loop 有效)
  - 注:这个 tool 的"真实人机界面"在 M2 完整 agent loop 里替换为更友好的形态
- `tools/state.py` 扩展:
  - `write_decision(item_id?, decision_type, rationale, action_taken?) → decision_id` tool
  - rationale 非空强制
  - `update_context_brief(brief: str) → brief_id` tool
  - 长度 ≤ 1000 token,超出报错
- `contacts.yaml` 加载(支持 aliases 应对脏数据 `李坤(gitcode)` 等)
- `tools/__init__.py`:`TOOL_REGISTRY` + `TOOL_SCHEMAS`(供 M2 直接喂进 Anthropic SDK 的 `tools=` 参数)

## 设计要点(强约束)
1. **P2 安全边界硬约束在 `send_welink` 工具内部实现**,绝不能依赖 agent 自觉。LLM 即使疯了猛调 `send_welink`,工具层也会全部 blocked。
2. **P4 决策必须留 rationale**:`write_decision` 在函数入口校验 rationale 非空且 ≥ 20 字符,否则抛错。
3. **TOOL_SCHEMAS 严格遵循 Anthropic Tool Use 格式**(`name`/`description`/`input_schema`),M2 直接用。每个 schema 的 `description` 字段要写清楚:这个 tool 什么时候用、输入输出什么、有什么副作用——LLM 靠这个判断要不要调。
4. **MockNotifier 的 jsonl 输出**字段对齐 `follow_up_log` 表结构,M4 切换真 CLI 时可以 diff 验证。
5. **ask_human 的 stdin 交互**是 MVP 临时方案。M2 集成 agent loop 后,可能换成更结构化的交互(打印 markdown 候选清单 + 解析 PM 自然语言回应等)。本 issue 先保证可用即可。
6. **不引入 LLM 调用** — 还是纯工具层。

## 输入 / 输出
**输入:** M1-A 产出的 WorkItem 列表 + `contacts.yaml` + `pm-agent.yaml`
**输出:**
- 一个集成测试脚本 `scripts/dry_run_pipeline.py`:不经过 agent,直接串联 `read_excel → query_rule_suggestions → ask_human(自动 yes)→ gen_message → send_welink(mock)→ write_decision`,验证 5 个 tool 协作正常

## 验收标准
1. ✅ `query_rule_suggestions` 输出 ≈ 77 条建议(与活跃候选数对齐 ±5)
2. ✅ 4 类消息模板都能正确渲染,占位符空值显示"未填写"
3. ✅ `send_welink` 在 5 种 blocked 场景下分别返回正确的 reason
4. ✅ 缺 `confirmation_token` 直接调 `send_welink` → `error: no_confirmation_token`
5. ✅ `MockNotifier` 写入 `state/mock_sent.jsonl`,字段对齐 `follow_up_log`
6. ✅ `contacts.yaml` 的 `aliases: [李坤(gitcode)]` 能解析回 `李坤` 的 welink_id
7. ✅ `write_decision` 拒绝空 rationale / 短于 20 字
8. ✅ 集成测试脚本一遍跑通,主 Excel mtime 不变,db 内 follow_up_log + decision_log 各有记录
9. ✅ `TOOL_SCHEMAS` 通过 `anthropic.types.ToolParam` 类型校验(可直接喂给 SDK)
