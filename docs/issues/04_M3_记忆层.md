# [M3] 记忆层:context_brief + decision_log 跨周期上下文加载

## 目标
让 agent 有"记性"。每次 wake 启动时,自动加载上次 brief + 最近决策历史 + 最近发送历史,作为初始 user message 的一部分喂入。**完成本 issue 后,连续跑 3 次 wake,第 3 次的 agent 能引用前 2 次的判断**——这才是真正的 agentic 长期助理。

## 范围
- `pm_agent/memory/loader.py`:
  - `load_initial_context(trigger_reason, lookback_days=14)` 函数
  - 读 `context_brief` 最新 1 条 → `prior_brief`
  - 读 `decision_log` 最近 N 条(N 可配置,默认 20)→ `recent_decisions`
  - 读 `follow_up_log` 最近 N 条 → `recent_sends`
  - 读 `item_state` 中所有 `vanished_at` 不为空的 → `vanished_items`
- `prompts/initial_user.md.j2` 扩展:
  - 加入上述四块字段
  - 用 Jinja 模板渲染,空值不展示对应段落
- `tools/state.py`:
  - `query_history(item_id, max_records=10) → list` tool — agent 在 loop 内可主动查某事项的历史
  - 这与 `query_state` 的区别:`query_state` 返回当前状态摘要,`query_history` 返回时间序列
- Brief 内容约束(在 `update_context_brief` 工具描述里强化):
  - 不超过 1000 token
  - 必须包含:本次扫描总览(总数/活跃数)、本次关键决策、未决问题、下次 wake 时应关注的重点
  - 推荐 markdown 结构,便于下次注入 prompt 时人/LLM 都易读
- Brief 历史保留策略:
  - `context_brief` 表保留最近 30 条
  - 超出自动归档至 `state/archive/briefs/{date}.md`
- 决策回顾命令:
  - `python -m pm_agent retrospect --days 7` — 打印近 7 天 decision_log + brief 摘要(给 PM 看,不调 LLM)

## 设计要点(强约束)
1. **记忆是单向的**:agent 只读旧 brief/decision,**不能修改它们**。decision_log append-only,brief 每次 insert 新行(不 UPDATE)。这保证审计完整性。
2. **prior_brief 注入位置**:在 initial user message 里,标记为"上次结束时的状态"。system prompt 不变,**保持稳定**(避免 prompt 漂移)。
3. **token 预算分配**:lookback 加载的内容 + system prompt 总 token 不超过 8k(留出 42k 给本次 loop)。超出时按以下顺序裁剪:recent_sends > recent_decisions > prior_brief(brief 永远保留)。
4. **vanished_items 必须在 prompt 里告知**:这是上次扫到、本次没扫到的事项,可能是 PM 改了原文或删了,agent 应该主动提及让 PM 确认。
5. **decision_log 不只记 LLM 的决策**,也记 PM 的拒绝/修改。例如 PM 在 ask_human 阶段否决了某条候选,agent 应调 `write_decision(decision_type='skip', rationale='PM 拒绝:...')`。

## 输入 / 输出
**输入:** wake 命令、已有的 SQLite 历史数据
**输出:**
- 初始 user message 包含完整上下文(prior brief + recent decisions + recent sends + vanished)
- agent 行为体现连续性(第 3 次能说"上次我们催了 X,这次他还没回")

## 验收标准
1. ✅ 连续 3 次 `wake`,第 3 次的 stderr 日志显示 prompt 含前 2 次 brief 摘要
2. ✅ 第 2 次 wake 时,agent 至少 1 次主动引用前次决策(可通过对话内容观察)
3. ✅ 故意删一条事项的原文 → 下次 wake 时 agent 主动询问 PM "这条事项不见了,是不是改了?"
4. ✅ 让前 N 条 decision_log 内容超大,验证裁剪逻辑:recent_sends 先被裁,brief 永远保留
5. ✅ `retrospect --days 7` 命令打印格式化的最近 7 天决策回顾,不调 LLM
6. ✅ `query_history(item_id)` 能返回某事项的完整历史(状态 + 催办 + 决策)时间序列
7. ✅ 30 条 brief 后,旧 brief 自动归档到 `state/archive/briefs/`,db 内只留 30 条
8. ✅ Token 预算超限时(模拟 20k lookback),裁剪后保证 + system prompt ≤ 8k
